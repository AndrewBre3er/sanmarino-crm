# AGENTS.md

## Роль этого файла

Это главный контекст для Codex и других coding agents в данном репозитории.

Перед любой работой агент обязан загрузить контекст по уровню риска задачи:

1. Для любого шага прочитать минимум:
   - `README.md`
   - `AGENTS.md`
   - `docs/ai/PROJECT_INDEX.md`
   - `docs/ai/CONTEXT_BRIEF.md`
   - `docs/ai/BUSINESS_RULES.md`
   - `docs/ai/CODING_RULES.md`
2. Для task-driven работы прочитать конкретный task-файл из `docs/tasks/`
   и все canonical docs, явно перечисленные в task-файле.
3. Для доменных, schema, state-machine, security, permissions, finance,
   inventory, logistics, KPI, API/event-contract и cross-domain изменений
   прочитать релевантные canonical docs до кодинга.
4. Для новых этапов, спорной бизнес-логики, архитектурных конфликтов,
   неясного scope или риска нарушения инвариантов прочитать весь каталог
   `docs/` в numeric order.
5. Следовать бизнес-логике и ограничениям проекта.
6. Не изобретать доменные правила, которых нет в документации.

Если агент не уверен, какой уровень контекста достаточен, он обязан выбрать
более широкий уровень чтения. `docs/ai/*`, `docs/tasks/*` и `.agents/skills/*`
ускоряют навигацию, но не переопределяют accepted docs и `AGENTS.md`.

### Operational scenario checklist

Для задач по CRM, ролям, заказам, отгрузке, оплатам, возвратам, логистике,
финансам, KPI/workspace UX и пользовательским сценариям агент обязан также
свериться с `docs/40-crm-workflow-and-role-scenarios.md` как с кратким
операционным checklist на противоречия.

При этом:

- `docs/40-crm-workflow-and-role-scenarios.md` не является новым источником истины
  для бизнес-логики
- он не может переопределять canonical / accepted docs
- если новое правило логики нужно сделать обязательным, его нужно фиксировать
  в соответствующих canonical docs, а не только в `docs/40-*`
- при конфликте всегда приоритет у `AGENTS.md` и accepted docs

## 2. Источники истины

Система не должна хранить все факты внутри CRM.

Источники истины:

- CRM: lead, client, contact, deal
- Orders: order, order items, операционный статус заказа
- Inventory: остатки, резервы, движения, средняя себестоимость
- Payments: оплаты и возвраты денег
- Logistics: слоты, задачи доставки, загрузка, факт доставки
- Finance: доходы по cash basis, расходы, статьи затрат
- KPI: производный слой, не источник истины

## 3. Канонические правила

### 3.1 Коммерческий и операционный слои разделены

Цепочка процесса:
`Lead -> Deal -> Order(s) -> Fulfillment(s)`

### 3.2 Доход признаётся по оплате

Базовая логика:

- cash basis
- не по подтверждению заказа
- не по факту отгрузки

### 3.3 Оплата — параллельный процесс

Оплата может произойти:

- до исполнения
- во время исполнения
- после исполнения

### 3.4 Расход товара — только по факту исполнения

Подтверждение заказа:

- не создаёт расход
- может создать резерв
- может создать бронирование логистики

### 3.5 Возврат только через ReturnRequest

Нельзя:

- возвращать деньги без `ReturnRequest`
- возвращать товар без `ReturnRequest`

### 3.6 Резервы и мягкие блокировки имеют TTL

Жёсткий durable reservation:

- создаётся только после подтверждения заказа
- не создаёт расход
- должен автоматически сниматься при отмене
- должен контролироваться через регулярную проверку

На стадии `Draft` допускается только:

- short-lived soft lock
- pre-reserve с коротким TTL
- предварительная блокировка без признания товарного расхода и без окончательного подтверждения заказа

### 3.7 Все критические мутации идемпотентны

Каждая критическая операция обязана иметь `idempotencyKey`.

### 3.8 State machine обязательна

Статусы нельзя переводить произвольно.
Вне схемы допускается только admin override с аудитом.

### 3.9 Предварительная проверка обязательна

До подтверждения заказа система должна уметь:

- проверить наличие
- показать доступность
- показать доступные слоты
- показать конфликт по логистике
- при необходимости создать short-lived soft lock для защиты от гонки подтверждения

### 3.10 Один заказ может иметь несколько delivery task

Связь:

- `Order -> DeliveryTask = 1:N`

Следствие:

- delivery status заказа агрегируется из связанных delivery task
- сценарий `PartiallyDelivered` обязателен

### 3.11 Междоменные мутации должны быть атомарными или компенсируемыми

Подтверждение заказа не должно оставлять зависший резерв без согласованной логистики.
Допустимы только:

- единая транзакция БД с rollback
- Saga
- Transactional Outbox + compensating actions

### 3.12 Возвраты проходят через quarantine

Возвращённый товар по умолчанию не возвращается сразу в `available`.
Сначала:

- quarantine
- дефектовка
- ручной перевод в доступный остаток при положительном результате проверки

### 3.13 Критичные сущности не удаляются физически

Для `Order`, `Deal`, `Payment`, `ReturnRequest` разрешён только soft delete.

### 3.14 KPI не строятся тяжёлыми realtime JOIN

Live KPI:

- обновляются асинхронно
- читаются из агрегатов или cache
- не создают тяжёлую нагрузку на транзакционные таблицы

### 3.15 Цена продажи и себестоимость разделены

`OrderItem` хранит retail price и итог продажи.
Себестоимость живёт в inventory-логике и не подменяется ценой продажи.

## 4. Ограничения для агента

Агент не должен:

- смешивать домены в один модуль
- придумывать поля, статусы и процессы без фиксации в docs
- проектировать API в противоречии с state machine
- проектировать финансы в логике accrual, если активна cash basis
- делать durable reservation на стадии draft вместо short-lived soft lock
- делать расход по факту подтверждения заказа
- использовать KPI как источник истины
- связывать возврат напрямую с оплатой без `ReturnRequest`

## 5. Правила проектирования кода

### 5.1 Модульность

Монорепо проектируется по утверждённым контрактам каталогов:

- `apps/web`
- `apps/api`
- `apps/worker`
- `packages/ui`
- `packages/config`
- `packages/types`

Доменные модули реализуются внутри `apps/api` по документации и только после bootstrap-фазы.

### 5.2 Границы

Каждый модуль должен иметь:

- свои сущности
- свои use cases
- свои статусы
- свои инварианты
- явные входы и выходы

### 5.3 Документация перед кодом

Если логика домена не ясна, сначала обновляется `docs/*`, потом код.

### 5.4 Без галлюцинаций

Если информация отсутствует:

- пометь это как `TBD`
- не заполняй пробелы выдуманными решениями
