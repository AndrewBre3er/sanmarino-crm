# 08. Architecture Fixes & Critical Blockers

## Статус документа

**ОБЯЗАТЕЛЬНО К ПРОЧТЕНИЮ ПЕРЕД НАПИСАНИЕМ КОДА.**

Этот документ содержит критические исправления к базовым документам `01-system-logic.md` - `07-roles-and-access.md`.
Уязвимости были выявлены в ходе архитектурного аудита.
При проектировании ER-модели БД, API, очередей и бизнес-логики этот документ имеет **наивысший приоритет** в местах конфликта с ранними документами.

---

## 1. Критичные блокеры

### 1.1 Race condition при подтверждении заказа

В ранней версии логики на этапе `Draft` не создавался резерв, а лишь проверялась доступность.
В конкурентной среде это создаёт окно гонки между проверкой и подтверждением заказа.

Обязательные правила:
- при начале оформления корзины или при переходе к коммерчески значимому `Draft` должен использоваться **soft lock** или краткоживущий предварительный резерв
- soft lock должен иметь короткий TTL, ориентир `5-10 минут`
- durable reservation по складу создаётся только в момент успешной операции `Draft -> Confirmed`
- на этапе подтверждения заказа обязательно должна существовать обработка ошибки `OutOfStockError`
- при неуспешном подтверждении заказ остаётся или возвращается в `Draft`, а пользователь и оператор получают явное сообщение о конфликте наличия

---

### 1.2 Частичные доставки

Связь `1 Order = 1 DeliveryTask` признана ошибочной.

Обязательные правила:
- связь должна быть `1 Order = 1..* DeliveryTask`
- один заказ может порождать несколько задач доставки по разным позициям, датам, складам, машинам или типам груза
- статус `Order.deliveryStatus` должен быть агрегированным
- минимум должен поддерживаться сценарий `PartiallyDelivered`, пока не закрыты все связанные `DeliveryTask`

---

### 1.3 Распределённые транзакции и частичный отказ

Переход заказа в `Confirmed` требует координации минимум с доменами `Inventory` и `Logistics`.

Риск:
- резерв склада может создаться успешно
- бронирование логистики может упасть
- система получит неконсистентное состояние с зависшим резервом и неподтверждённой логистикой

Обязательные правила:
- междоменные мутации должны реализовываться через **Saga** и/или **Transactional Outbox**
- если система опирается на одну монолитную БД, допустима единая транзакция с полным `ROLLBACK`
- если логистика не подтверждена, резерв не должен оставаться активным без компенсирующего действия
- операция `OrderConfirmation` должна оставлять систему либо в полностью согласованном состоянии, либо в безопасно откатанном/компенсированном

---

## 2. Логические и операционные уязвимости

### 2.1 Карантин возвратов

Возвращённый технически сложный товар нельзя автоматически возвращать в доступный остаток.

Обязательные правила:
- в домене `Inventory` должен существовать статус остатка `quarantine` и/или отдельная виртуальная зона хранения
- движение `return_to_stock` по умолчанию должно помещать товар в `quarantine`
- перевод из `quarantine` в `available` возможен только после ручной дефектовки и подтверждения кладовщиком или уполномоченной ролью

---

### 2.2 Производительность Live KPI

Тяжёлые realtime `JOIN` между `CRM`, `Orders`, `Payments`, `Finance` и `Inventory` недопустимы для операционных виджетов и dashboard-блоков.

Обязательные правила:
- запретить тяжёлые realtime `JOIN` для пользовательских Live KPI
- live-метрики должны обновляться асинхронно по событиям
- источником для live-виджетов должны быть агрегированные таблицы, materialized-представления, precomputed snapshots и/или cache
- KPI остаётся read-layer и не может быть источником первичного бизнес-факта

---

## 3. Директивы для кодогенерации

### 3.1 Idempotency layer

- проверка `idempotencyKey` должна происходить **до** входа в доменные сервисы
- реализация должна существовать как middleware, guard, interceptor или эквивалентный pre-controller слой
- доменный сервис не должен узнавать о повторе только после побочных мутаций

### 3.2 Soft deletion

Следующие сущности запрещено удалять физически:
- `Order`
- `Deal`
- `Payment`
- `ReturnRequest`

Обязательные правила:
- использовать `deleted_at` и/или `is_deleted`
- физический `DELETE` для этих сущностей запрещён
- soft delete не должен ломать `reconciliation`, audit и отчётность

### 3.3 Price source of truth

При проектировании `OrderItem` необходимо жёстко разделять:
- базовую розничную цену продажи
- скидку
- зафиксированную итоговую цену строки
- себестоимость

Обязательные правила:
- розничная цена подтягивается из каталога и фиксируется в `OrderItem` при формировании `Draft`
- себестоимость не должна считаться retail-ценой
- фактическая себестоимость живёт в `Inventory` и рассчитывается по принятой модели стоимости, включая `weighted average`

## 4. Приоритет применения

Если любой ранний документ:
- говорит, что в `Draft` вообще не существует мягкой блокировки
- исходит из модели `1 Order = 1 DeliveryTask`
- допускает частичный междоменный успех без компенсации
- переводит возврат напрямую в `available`
- предполагает Live KPI через тяжёлые runtime `JOIN`
- допускает физическое удаление `Order`, `Deal`, `Payment`, `ReturnRequest`

то этот документ **переопределяет** такую логику.
