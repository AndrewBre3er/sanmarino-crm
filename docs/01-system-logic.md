# 01. System Logic

## Статус документа

Каноническая доменная логика проекта.
Любая БД, API и UI должны соответствовать этому документу.

---

## 1. Главные принципы

### 1.1 Коммерческий и операционный слой разделены
Цепочка:
`Lead -> Deal -> Order(s) -> Fulfillment(s)`

### 1.2 Доход признаётся по кассовому методу
Базовая логика:
- доход фиксируется по поступлению денег
- подтверждение заказа доход не создаёт
- отгрузка сама по себе доход не создаёт

### 1.3 Оплата — параллельный процесс
Оплата может быть:
- до исполнения
- во время исполнения
- после исполнения

### 1.4 Расход товара создаётся только по факту исполнения
Подтверждение заказа:
- не списывает товар
- может создать durable reservation
- может создать логистическое бронирование
- не может считаться успешным, если междоменные последствия не согласованы

### 1.5 Возврат — только через ReturnRequest
Все возвраты строятся от единой сущности `ReturnRequest`.

### 1.6 Резерв и мягкая блокировка не бесконечны
Durable reservation:
- создаётся только для подтверждённого заказа
- имеет TTL
- снимается автоматически или под аудитом

Soft lock / pre-reserve:
- допускается на этапе `Draft`
- нужен для защиты от race condition между проверкой и подтверждением
- имеет короткий TTL, ориентир `5-10 минут`
- не является товарным расходом и не заменяет полноценный reservation

### 1.7 KPI — производный слой
KPI нельзя использовать как источник истины.
Live KPI должны обновляться асинхронно по событиям и читаться из агрегатов или cache, а не через тяжёлые runtime JOIN по транзакционным таблицам.

## 4. Важные запреты

Система не должна допускать:
- двойную оплату одного события
- двойное списание товара
- возврат денег без `ReturnRequest`
- возврат товара без `ReturnRequest`
- резерв на стадии draft
- произвольный перевод статусов
- признание дохода не по cash basis
- существование заказа без сделки
- выполнение доставки без `DeliveryTask`


### 1.8 Один заказ может иметь несколько delivery task
Связь `Order -> DeliveryTask` является отношением `1:N`.
Агрегированный статус доставки заказа должен вычисляться по связанным задачам доставки.

### 1.9 Подтверждение заказа — междоменная операция
Операция `Draft -> Confirmed` должна:
- либо выполняться в рамках одной транзакции БД с полным rollback
- либо использовать Saga / Transactional Outbox и компенсирующие действия

Частичный успех без компенсации запрещён.

### 1.10 Возвратный товар по умолчанию попадает в quarantine
Возврат через `ReturnRequest` не должен автоматически увеличивать доступный остаток.
Сначала товар поступает в `quarantine`, затем после дефектовки может быть переведён в `available`.

### 1.11 Факты продажи и себестоимости разделены
`OrderItem` хранит цену продажи.
Себестоимость живёт в домене Inventory и рассчитывается по принятой модели стоимости. Цена продажи не может подменять cost.
