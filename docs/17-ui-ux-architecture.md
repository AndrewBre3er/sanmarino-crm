# 17. UI/UX Architecture

## 1. Назначение документа

Документ фиксирует каноническую UI/UX-архитектуру CRM/ERP-системы.

Цель документа:
- задать логику интерфейса до начала визуальной реализации
- определить структуру экранов и навигации
- зафиксировать ролевую модель интерфейса
- исключить проектирование "одного общего UI для всех"
- дать агенту и разработчику единый источник истины для дальнейшей сборки экранов

Этот документ описывает не визуальный стиль бренда, а логику интерфейса продукта.

---

## 2. Базовый принцип

Система должна иметь **единое приложение**, но **не единый одинаковый интерфейс**.

Каждый отдел работает в собственной рабочей области.

Это означает:
- у ролей разный левый сайдбар
- у ролей разные стартовые страницы
- у ролей разный состав списков, карточек, виджетов и отчётов
- у ролей разные доступные массовые действия
- у ролей разные поля в формах и карточках
- часть данных должна быть не только недоступна для редактирования, но полностью скрыта из интерфейса

Следовательно, UI строится по модели:

`Application Shell -> Role Workspace -> Domain Screens -> Object Views -> Actions`

Дополнительное правило текущего v1:
- названия ролей в интерфейсе показываются на русском языке
- техническая авторизация использует канонические role codes:
  - `admin`, `seller`, `warehouse`, `logistics`, `finance`, `ceo`
  - optional: `driver`, `marketing`
- marketing workspace не является обязательным стартовым workspace
- executive dashboard обязан разделять денежные показатели, отгрузку и контрольные денежные потоки от водителей
- `Supplier Requests` является обязательным MVP экраном (list + card) для `seller` и `warehouse`

---

## 3. Цели UI/UX системы

Интерфейс должен:
- ускорять ежедневные операционные действия
- снижать количество ошибок при переводе статусов и ручных изменениях
- показывать сотруднику только релевантные его роли данные
- не перегружать пользователя междоменной информацией
- поддерживать сквозной сценарий от лида до исполнения заказа
- отдельно поддерживать контроль, аналитику и аудит для руководителя

Интерфейс не должен:
- пытаться показать всю систему всем сотрудникам одновременно
- смешивать коммерческий, складской, финансовый и логистический слои на одном экране без необходимости
- раскрывать чувствительные показатели нецелевым ролям
- позволять пользователю обходить state machine через свободные действия

## 4. Канонический паттерн приложения

### 4.1 Верхний уровень

Любой экран строится из 5 слоёв:

1. **App Shell**
   - верхняя панель
   - навигация
   - глобальный поиск
   - уведомления
   - пользовательское меню

2. **Workspace Layer**
   - роль
   - отдел
   - стартовый dashboard
   - список разрешённых доменов

3. **Module Layer**
   - CRM
   - Orders
   - Inventory
   - Payments
   - Logistics
   - Finance
   - KPI
   - Audit
   - Users

4. **Screen Layer**
   - список
   - канбан
   - календарь
   - карточка
   - журнал
   - отчёт

5. **Action Layer**
   - создать
   - подтвердить
   - зарезервировать
   - оплатить
   - отгрузить
   - назначить доставку
   - оформить возврат
   - выполнить сверку

## 5. Основные UX-принципы

### 5.1 Role-first

Сначала определяется роль пользователя, затем система показывает его рабочее пространство.

Не допускается сценарий, где пользователь после входа попадает в общий перегруженный экран со всеми модулями.

### 5.2 Object-first

Пользователь работает не с абстрактными экранами, а с объектами:
- lead
- deal
- order
- payment
- delivery task
- stock movement
- return request

Карточка объекта должна быть основным узлом работы.

### 5.3 Status-first

Следующее действие должно зависеть от статуса объекта.

Интерфейс не должен показывать все действия сразу. Он должен показывать только допустимые действия в рамках state machine.

### 5.4 Visibility by necessity

Если данные не нужны роли для принятия решения, данные должны быть скрыты.

Не рекомендуется замещать скрытие пассивным read-only режимом, если сама информация для роли избыточна.

### 5.5 Progressive disclosure

Сложная информация раскрывается по слоям:
- сначала ключевой статус
- потом основные атрибуты
- потом связанные факты
- потом историю и аудит

### 5.6 Cross-domain traceability

На карточке объекта должны быть видны связанные сущности из других доменов, но в пределах прав роли:
- из deal видно связанные orders
- из order видно оплаты, резерв, доставку, возвраты
- из logistics task видно order и клиента
- из payment видно order/deal linkage

### 5.7 Safe mutations

Любое критическое действие должно сопровождаться:
- объяснением последствий
- проверкой предусловий
- подтверждением действия
- записью в аудит

## 6. Общая информационная архитектура

### 6.1 Глобальные разделы приложения

Канонические разделы:
- Home
- CRM
- Orders
- Inventory
- Payments
- Logistics
- Finance
- KPI
- Audit
- Administration

Показываются не все разделы, а только разрешённые рабочему пространству роли.

### 6.2 Глобальный поиск

Глобальный поиск должен искать:
- клиента
- контакт
- deal
- order
- payment
- delivery
- sku
- return request

Поиск обязан учитывать права доступа.

### 6.3 Унифицированные экранные паттерны

Для каждой сущности используются одинаковые паттерны:
- список
- карточка
- таймлайн
- связанные объекты
- действия
- история изменений

Это снижает когнитивную нагрузку между доменами.

---

## 7. Каноническая структура списка

Любой list screen должен содержать:
- заголовок экрана
- счётчики по статусам
- быстрые фильтры
- расширенные фильтры
- поиск
- сортировку
- таблицу или канбан
- массовые действия в рамках прав
- экспорт, если он разрешён

Обязательные UX-требования:
- pinned filters for daily work
- сохранённые представления
- видимые системные статусы
- быстрый переход в карточку
- индикаторы блокировок, конфликтов, просрочек и ошибок

---

## 8. Каноническая структура карточки объекта

Карточка любого объекта должна иметь 6 зон:

1. **Identity**
   - id
   - тип объекта
   - статус
   - приоритет
   - ответственный

2. **Primary Summary**
   - ключевые данные для быстрого чтения

3. **Allowed Actions**
   - только допустимые действия

4. **Related Objects**
   - связи с другими доменами

5. **Timeline**
   - события по объекту

6. **Audit / System Info**
   - кто изменил
   - когда изменил
   - override / system flags

Запрещено прятать статус объекта внутри второстепенных вкладок.

## 10. Стартовые экраны по ролям

После входа пользователь попадает не на общий dashboard, а на **role home**.

### Продавец Home (`seller`)
Показывает:
- новые лиды
- lead, ожидающие перевода в `в обработке`
- активные `Deal`
- supplier coverage status
- заказы, требующие резерва / partial reserve / supplier coverage
- заказы, требующие реакции менеджера

### Маркетинг Home (`marketing`, optional later)
Показывает:
- лиды по каналам
- CPL/CAC/ROMI
- динамику качества лидов
- расходы по маркетингу
- аномалии каналов

### Финансист Home (`finance`)
Показывает:
- входящие оплаты
- невязки по платежам
- возвраты
- расходы
- supplier payables
- деньги у водителей
- критические расхождения сверки

### Логист Home (`logistics`)
Показывает:
- сегодняшний календарь доставок
- перегруженные интервалы
- неподтверждённые задачи
- срывы и переносы
- доступность водителей
- проблемные заказы по просроченным деньгам

### Кладовщик Home (`warehouse`)
Показывает:
- низкие остатки
- активные резервы
- partial reserve / full reserve readiness
- ожидаемые приходы
- supplier request со статусом `received_with_discrepancy`
- проблемные отгрузки

### Исполнительный директор Home (`ceo`)
Показывает:
- сквозные KPI
- узкие места по воронке
- денежную выручку отдельно от `отгружено`
- деньги у водителей
- supplier payables
- логистическую нагрузку
- остатки и дефицит
- междоменные расхождения
- критические override и ошибки данных

### Водитель Home (`driver`)
Показывает:
- маршрут на сегодня
- список задач
- адреса
- окна времени
- контакты клиента
- статус выполнения
- фиксацию результата доставки

## 12. Модель уведомлений

Уведомления должны быть контекстными и ролевыми.

Примеры:
- `seller`: новый lead, просроченный follow-up, deal без следующего шага
- `finance`: неопознанный платёж, возврат в ожидании, кассовая невязка
- `logistics`: конфликт слота, срыв доставки, деньги от водителя не подтверждены в срок
- `warehouse`: частичный резерв, отрицательный остаток, конфликт отгрузки, расхождение при приёмке поставки
- `ceo`: критическая невязка сверки, проблемный заказ, деньги у водителя, сбой интеграции, системный риск

Уведомления не должны дублировать весь event stream.
Они должны показывать только то, что требует решения пользователя.

## 13. UX-правила для междоменных связей

### 13.1 Deal -> Order
На карточке deal должны быть видны:
- связанные orders
- supplier request
- reserve coverage summary
- статус каждого order
- сумма
- оплата в агрегированном виде
- прогресс исполнения

### 13.2 Order -> Operations
На карточке order должны быть видны:
- items
- резерв
- supplier coverage
- платежи
- доставка / самовывоз
- fulfillment status
- control flags (`OnControl` / `Problem`)
- возвраты
- timeline

### 13.3 Payment -> Context
На карточке payment должны быть видны:
- связанный order или deal
- статус распознавания и проведения
- сумма
- канал
- возвраты / корректировки

### 13.4 Logistics Task -> Context
На карточке логистической задачи должны быть видны:
- клиент
- адрес
- order
- состав доставки
- временное окно
- водитель
- исход выполнения

## 14. UX-ограничения по чувствительным данным

Интерфейс обязан поддерживать как минимум 4 уровня скрытия:

1. **Hidden**
   - блок отсутствует полностью

2. **Visible Read-only**
   - пользователь видит, но не редактирует

3. **Visible Action-limited**
   - пользователь может видеть и делать только часть операций

4. **Visible Full**
   - полная видимость и разрешённые операции

Это правило применяется к:
- марже
- себестоимости
- закрытым расходам
- управленческим KPI
- override-операциям
- audit-журналам
- экспортам

---

## 15. Канонические формы

Форма должна строиться по правилам:
- группировка по смысловым секциям
- системные поля отдельно от пользовательских
- обязательные поля видимы явно
- зависимые поля раскрываются по условиям
- опасные действия вынесены в отдельную зону
- сохранение черновика допустимо там, где есть draft lifecycle

Формы не должны содержать поля из соседних доменов без явной причины.

---

## 16. Ошибки и системные состояния

UI обязан показывать:
- пустое состояние
- загрузку
- частичную недоступность данных
- блокировку действия из-за state machine
- конфликт данных
- результат idempotent retry
- предупреждение об устаревшем состоянии карточки

Текст ошибок должен объяснять:
- что произошло
- почему действие недоступно
- что делать дальше

---

## 17. Мобильность и адаптивность

На старте система проектируется как desktop-first backoffice.

Обязательные mobile-first исключения:
- интерфейс водителя
- быстрые подтверждения логистики
- критические уведомления для ответственных ролей

Основные рабочие места backoffice оптимизируются под desktop / laptop.

---

## 18. Что не фиксируется этим документом

Документ намеренно не фиксирует:
- цветовую систему бренда
- UI-kit библиотеки
- frontend framework
- конкретный компонентный стек
- конкретные брейкпоинты

Это определяется отдельным документом реализации.

---

## 19. Итоговое правило

UI/UX проекта должен проектироваться как **система ролевых рабочих пространств**, а не как одна общая панель с одинаковой навигацией для всех.

Главная единица проектирования интерфейса:

`роль -> рабочее пространство -> объект -> допустимое действие`
