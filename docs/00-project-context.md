# 00. Project Context

## Статус документа

Этот файл фиксирует только тот уровень контекста, который должен быть постоянным для всего проекта.

---

## 1. Назначение проекта

Проект — внутренняя CRM/ERP система для бизнеса СанМарино.

Система должна покрывать единый цикл:
- работа с лидом
- ведение сделки
- формирование заказа
- резервирование и движение товара
- оплата и возврат денег
- доставка или самовывоз
- финансовый учёт
- KPI и контроль расхождений

---

## 2. Что система должна объединять

Система объединяет следующие контуры:
- CRM
- Orders
- Inventory
- Payments
- Logistics
- Finance
- KPI / Analytics
- Users / Roles / Permissions
- Audit
- Reconciliation

---

## 3. Архитектурная идея

CRM является центральным оркестратором процесса, но не является источником истины для всех фактов.

Источники истины распределены между доменами:
- коммерческий контекст — CRM
- операционное исполнение — Orders
- товарные факты — Inventory
- денежные факты — Payments
- логистические факты — Logistics
- финансовое отражение — Finance
- аналитика — KPI как производный слой

---

## 4. Каноническая процессная цепочка

`Lead -> Deal -> Order(s) -> Fulfillment(s)`

Смысл:
- `Lead` — первичный вход
- `Deal` — коммерческий контейнер
- `Order` — операционный объект
- `Fulfillment` — факт исполнения

Одна сделка может порождать несколько заказов.

---

## 5. Что документ сознательно не фиксирует

Этот контекст не фиксирует:
- физическую схему БД
- конкретный backend-стек
- frontend-стек
- формат API
- очереди и брокеры
- CI/CD реализацию
- VPS-стек выполнения

Эти решения должны приниматься позже отдельными документами.
