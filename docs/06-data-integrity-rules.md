# 06. Data Integrity Rules

## 1. Idempotency

Каждая критическая мутация должна иметь `idempotencyKey`.
Проверка ключа должна выполняться до входа запроса в доменный сервис.

Минимум:
- payment create
- refund create
- inventory receipt
- inventory issue
- supplier request confirmation
- critical order transition
- shipment / fulfillment confirmation

Правило:
- повтор с тем же ключом не должен создавать второй результат

---

## 2. Soft lock, reservation и auto-order rule

Short-lived soft lock:
- допускается на стадии коммерческой подготовки
- используется для защиты от race condition
- имеет короткий TTL
- не заменяет durable reservation

Durable reservation:
- существует только для `Order`
- не должна жить напрямую на `Deal`
- снимается при отмене или освобождении заказа
- контролируется регулярной задачей
- ручное снятие попадает в audit

Если пользователь инициирует резерв из `Deal`, система должна:
- автоматически создать `Order`
- автоматически создать `Reservation`
- сделать это как одну согласованную операцию

---

## 3. State Machine Enforcement

Любой переход статуса:
- валидируется на backend
- не должен зависеть только от UI
- вне схемы разрешён только через admin override с audit

Control-флаги `OnControl` и `Problem` не заменяют основной статус заказа.
Это overlay-поля поверх операционного состояния.

---

## 4. Source of Truth Rule

Нельзя:
- считать факт оплаты из `Order` без `Payments`
- считать факт доставки из `Deal` без `Logistics`
- считать остаток из CRM
- считать KPI первичным фактом
- считать подтверждённую выручку из сообщения водителя без денежного подтверждения

---

## 5. Междоменная атомарность и компенсация

Критические операции, затрагивающие несколько доменов:
- должны выполняться в одной БД-транзакции с rollback
- либо через Saga / Transactional Outbox + compensating actions

Нельзя оставлять:
- активный reservation без `Order`
- частично созданный order без согласованного reservation/supply-state
- незавершённые междоменные мутации без следа в audit/system events

---

## 6. Soft delete

Следующие сущности не удаляются физически:
- `Order`
- `Deal`
- `Payment`
- `ReturnRequest`

Для них допустимы только:
- `deleted_at`
- `is_deleted`
- архивный/отменённый статус, если применимо

---

## 7. Деньги от водителя и контрольные сроки

Если заказ отгружен, а деньги ещё не подтверждены:
- заказ получает `OnControl`
- деньги водителя считаются отдельным контрольным денежным потоком, а не подтверждённым остатком денег

Если деньги не подтверждены до следующего рабочего дня:
- заказ получает `Problem`
- ответственный контур — логистика
- эскалация идёт директору

Снять `Problem` можно только после подтверждения поступления денег финансистом или директором.

---

## 8. Возвратный quarantine

Любой товарный возврат:
- по умолчанию зачисляется в `quarantine`
- не увеличивает `available` автоматически
- переводится в доступный остаток только после ручной дефектовки

---

## 9. Live KPI

Live KPI:
- не должны строиться тяжёлыми runtime JOIN по транзакционным таблицам
- должны обновляться асинхронно по событиям
- должны читаться из агрегированных таблиц, materialized views или cache

---

## 10. Price and cost source of truth

В `OrderItem` должны быть разделены:
- retail price
- скидка
- line total
- cost

Себестоимость не может храниться как retail price и не должна подменять ценовую логику заказа.

---

## 11. Reconciliation

Ежедневно должна запускаться сверка:

### Orders ↔ Payments
Подтверждённые оплаты должны совпадать с денежными операциями.

### Orders ↔ Driver money control
Заказы `OnControl` и `Problem` должны сверяться с неподтверждёнными денежными поступлениями от водителей.

### Orders ↔ Inventory
Отгруженные позиции должны совпадать с расходными движениями товара.

### Inventory ↔ Finance
Поступления и закупочные расходы должны быть сопоставимы.

### Logistics ↔ Orders
Delivery/self-pickup операции должны совпадать с фактом отгрузки и закрытия заказа.

Результат:
- reconciliation report
- alert ответственному при выходе за допуск
