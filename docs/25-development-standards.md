# 25. Development Standards


## Purpose

Документ фиксирует обязательные правила разработки до начала активного кодинга.

## Основные правила

- Если доменное правило неясно, сначала обновляется `docs/`, потом код.
- Нельзя добавлять новые статусы, переходы и критичные поля без документации.
- Нельзя прятать важную бизнес-логику в UI или случайных helper-файлах.
- Код должен укладываться в доменные границы.
- Если решение не утверждено, используется `TBD`, а не вымышленная реализация.

## Repo hygiene

Обязательно:
- осмысленные имена файлов и каталогов
- отсутствие мусорных debug-файлов
- отсутствие секретов в репозитории
- обновление документации при изменении логики

## Branching and commits

Рекомендуется:
- `main` — стабильная ветка
- feature branches — отдельные задачи
- hotfix branches — срочные исправления

Рекомендуемый формат коммитов:
- `feat: ...`
- `fix: ...`
- `docs: ...`
- `refactor: ...`
- `test: ...`
- `chore: ...`

## Структура кода

Внутри домена желательно разделять:
- entities / models
- use cases / services
- repositories / adapters
- policies / permissions
- validation
- tests

Точная структура зависит от утверждённого стека.

## API and contract discipline

- Контракты должны быть явными.
- Недопустимые действия должны возвращать отказ, а не silent success.
- Idempotent operations должны иметь явный механизм.
- При изменении контракта обновляются `14-api-contracts.md` и тесты.

## Migration discipline

- миграция воспроизводима
- не ломает production data незаметно
- поставляется согласованно с кодом
- high-risk migrations проверяются отдельно

## Testing discipline

Каждое исправление бага должно приводить к:
- добавлению или обновлению теста
- фиксации сценария воспроизведения
- повторной проверке затронутого участка

## Definition of done

Задача завершена, если:
- логика реализована по документации
- код укладывается в доменные границы
- тесты обновлены и проходят
- права доступа не нарушены
- документация обновлена при необходимости
- нет очевидных регрессий по core flow

## Codex working rules

Codex должен:
- читать `AGENTS.md` и `docs/` перед началом
- не менять широкий объём файлов без необходимости
- сначала исправлять логику, потом polish
- указывать `TBD`, если решение не утверждено
- не придумывать интеграции и инфраструктуру


## v8 Architecture Overrides

- Нельзя реализовывать физический `DELETE` для `Order`, `Deal`, `Payment`, `ReturnRequest`.
- Нельзя проектировать `Order -> DeliveryTask` как `1:1`.
- Нельзя переводить возвратный товар сразу в `available` без `quarantine`.
- Нельзя проверять idempotency только внутри доменного сервиса после побочных мутаций.
