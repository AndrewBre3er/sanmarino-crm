# 05. Process Flows

## 1. Продажа с доставкой

### Шаг 1. Lead
Создаётся lead.

### Шаг 2. Deal
Lead конвертируется в deal.
Менеджер ведёт коммерческое согласование.

### Шаг 3. Draft Order
Менеджер собирает черновик:
- товары
- дата
- способ исполнения

Система показывает:
- наличие
- доступность
- свободные слоты
- конфликты

На этом шаге:
- durable reservation ещё не создаётся
- может быть создан short-lived soft lock / pre-reserve
- мягкая блокировка должна иметь короткий TTL
- слот может проходить предварительную проверку, но окончательное бронирование должно быть частью confirm-операции

### Шаг 4. Confirmed Order
После подтверждения выполняется согласованная операция `OrderConfirmation`.

Обязательные эффекты:
- заказ переводится в `Confirmed`
- создаётся durable reservation
- бронируется логистический слот или эквивалентное окно исполнения
- создаётся одна или несколько delivery task, если режим исполнения = delivery

Обязательные гарантии:
- при нехватке товара система возвращает `OutOfStockError`
- при отказе логистики операция не должна оставлять зависший резерв
- допустимы только единая БД-транзакция с rollback либо Saga / Transactional Outbox с compensating action

### Шаг 5. Payment
При оплате:
- создаётся payment
- создаётся cash operation
- finance признаёт доход по cash basis

### Шаг 6. Fulfillment
При фактическом исполнении:
- подтверждается одна delivery task или факт выдачи
- создаётся issue в inventory только по реально исполненной части
- обновляется статус исполнения
- агрегированный `Order.deliveryStatus` пересчитывается по всем связанным delivery task

Если доставка частичная:
- заказ остаётся в операционно незавершённом состоянии
- `Order.deliveryStatus` может стать `PartiallyDelivered`
- последующие delivery task продолжают цикл до полного закрытия заказа

### Шаг 7. Close
После исполнения и урегулирования денежных последствий заказ закрывается.

## 2. Продажа с самовывозом

Отличия:
- вместо delivery slot используется pickup window
- delivery task не создаётся
- расход создаётся по факту выдачи

---

## 3. Закупка

Поток:
1. inventory фиксирует потребность
2. оформляется закупка
3. при поступлении создаётся purchase receipt
4. создаётся receipt movement
5. пересчитывается средняя себестоимость
6. finance отражает расход

---

## 4. Возврат

Поток:
1. создаётся `ReturnRequest`
2. фиксируются позиции, количество, причина
3. принимается решение
4. создаются последствия:
   - возврат денег
   - возврат товара
   - списание
5. возвращённый товар по умолчанию попадает в `quarantine`
6. только после дефектовки товар может быть переведён в `available`

Прямой возврат товара сразу в доступный остаток запрещён.
