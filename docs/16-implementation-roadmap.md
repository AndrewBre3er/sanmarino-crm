# 16. Implementation Roadmap

## Статус документа

Канонический технический документ для поэтапной реализации проекта.

Документ определяет:
- порядок реализации
- зависимости между этапами
- минимальный MVP
- критерии завершения этапов
- границы того, что можно откладывать без нарушения логики проекта

Документ опирается на уже утверждённые правила проекта и не должен им противоречить:
- `Lead -> Deal -> Order(s) -> Fulfillment(s)`
- `cashBasis`
- `ReturnRequest`
- `state machine`
- `idempotency`
- `reconciliation`
- разделение доменов и источников истины

Этот документ **не является** детальным sprint backlog.
Это управленческий и архитектурный порядок сборки системы.

---

## 1. Цель дорожной карты

Цель — реализовать систему без хаотичного наращивания функций.

Порядок работ должен:
- сначала фиксировать каноническую логику
- затем создавать первичные домены
- затем включать критические транзакционные сценарии
- только после этого строить производные KPI, автоматизации и вторичные удобства

Ключевой принцип:
**сначала источник истины и инварианты, потом автоматизация и аналитические надстройки.**

---

## 2. Общая стратегия реализации

Реализация делится на 8 этапов:

1. Foundation и каркас проекта
2. CRM домен
3. Orders домен
4. Inventory домен
5. Payments + Finance
6. Logistics + Fulfillment
7. Returns + Reconciliation + Audit hardening
8. KPI / Reporting / Automation layer

Такой порядок нужен, потому что:
- без CRM неоткуда брать lead/deal
- без Order нельзя корректно связать оплату, склад и логистику
- без Inventory нельзя гарантировать корректный резерв и расход
- без Payments/Finance нельзя реализовать cash basis
- без Logistics/Fulfillment нельзя корректно привязать расход к факту исполнения
- без Returns/Reconciliation система остаётся уязвимой к расхождениям
- KPI строятся только после появления первичных фактов

---

## 3. Этап 0. Project Foundation

## 3.1 Цель

Подготовить среду, репозиторий и инженерный каркас, на который дальше можно безопасно накладывать доменную логику.

---

## 3.2 Состав работ

- создать репозиторий и базовую структуру папок
- зафиксировать `AGENTS.md` и пакет `docs/*.md`
- настроить локальную среду Win11 + Git Bash + VS Code
- подготовить `.env.example`
- подготовить базовые правила git / editor / line endings
- подготовить заготовки для миграций, тестов, scripts и deploy
- определить стратегию конфигурации для dev/stage/prod — без раскрытия секретов

---

## 3.3 Результат этапа

На выходе должны существовать:
- читаемый контекст проекта
- воспроизводимая локальная среда
- базовая структура репозитория
- единые правила работы для Codex и разработчика

---

## 3.4 Done criteria

- проект открывается и воспроизводится локально
- контекстные документы лежат в репозитории
- секреты не закоммичены
- деплой-каркас на VPS описан и отделён от локальной среды

---

## 4. Этап 1. Canonical Data Model

## 4.1 Цель

Сначала зафиксировать основу: БД-модель, контракты API и событийную модель, чтобы код не рос на предположениях.

---

## 4.2 Состав работ

- утвердить `13-database-architecture.md`
- утвердить `14-api-contracts.md`
- утвердить `15-event-model.md`
- определить стратегию миграций
- определить соглашения по ID, timestamps, soft delete, audit references
- подготовить первую волну схем БД без прикладной избыточности

---

## 4.3 Результат этапа

- есть каноническая схема сущностей
- есть API-уровень ресурсов и команд
- есть карта событий
- можно начинать код без риска, что базовые домены будут собраны вразнобой

---

## 4.4 Done criteria

- все основные сущности имеют зафиксированную логическую модель
- все критические переходы описаны command-style API
- все критические события имеют канонические имена и payload
- не осталось противоречий между логикой, БД и API

---

## 5. Этап 2. CRM MVP

## 5.1 Цель

Сделать первичный входящий контур: lead, client, contact, deal.

---

## 5.2 Обязательные сущности

- `Lead`
- `Client`
- `Contact`
- `Deal`

---

## 5.3 Обязательные функции

- создание lead
- просмотр списка и карточки lead
- конвертация lead в deal
- создание / редактирование client и contact
- перевод deal по state machine
- фиксация `deal.won` / `deal.lost`
- аудит ключевых изменений

---

## 5.4 Что пока не делать

Не добавлять на этом этапе:
- деньги как часть CRM
- складские остатки в CRM
- произвольные кастомные статусы вне state machine
- тяжёлые маркетинговые отчёты

---

## 5.5 Done criteria

- lead и deal живут как первичные CRM-сущности
- конвертация в deal работает воспроизводимо
- deal можно довести до `Won` или `Lost`
- все критические изменения попадают в audit

---

## 6. Этап 3. Orders MVP

## 6.1 Цель

Ввести управляемый контур заказа, связанный со сделкой.

---

## 6.2 Обязательные сущности

- `Order`
- `OrderItem`
- `Fulfillment`

---

## 6.3 Обязательные функции

- создание draft order только из deal
- редактирование состава заказа
- подтверждение заказа командой `confirm`
- отмена заказа
- базовые статусы по state machine
- связка order ↔ deal
- создание событий `order.created`, `order.confirmed`, `order.cancelled`

---

## 6.4 Критические ограничения

- order без deal запрещён
- `Draft` не создаёт резерв
- order не признаёт оплату сам по себе
- order не списывает товар сам по себе

---

## 6.5 Done criteria

- order создаётся только в допустимом контуре
- переходы по статусам валидируются backend-слоем
- события по order публикуются после фиксации факта
- нет смешения order-логики с payment и inventory

## 6.6 Mandatory pre-coding gate for extended order requirements

Before full implementation of extended order-commercial scenarios, `37-order-flow-pre-coding-requirements.md` must be applied in canonical docs.

This gate is mandatory for:
- installer/designer participant model
- separate invoice document
- suppliers and supplier request flow
- strict unit-of-measure catalog

---

## 7. Этап 4. Inventory MVP

## 7.1 Цель

Сделать склад как отдельный источник истины по остаткам, резервам и движениям.

---

## 7.2 Обязательные сущности

- `Product`
- `Warehouse`
- `StockBalance`
- `Reservation`
- `InventoryMovement`
- `PurchaseReceipt`

---

## 7.3 Обязательные функции

- просмотр остатков по складам
- создание резерва только для confirmed order
- TTL резервов
- ручное и автоматическое снятие резерва
- складской приход
- складской расход
- пересчёт средней себестоимости
- события `reservation.created`, `reservation.released`, `inventory.receipt.recorded`, `inventory.issue.recorded`

---

## 7.4 Критические ограничения

- нельзя править остаток напрямую из CRM
- журнал движений обязателен
- расход не должен происходить по подтверждению заказа
- резерв не бесконечный

---

## 7.5 Done criteria

- остатки воспроизводимы через движения
- резервы контролируются TTL
- расход оформляется только отдельным фактом
- доступны минимальные сверки order ↔ inventory

---

## 8. Этап 5. Payments + Finance MVP

## 8.1 Цель

Реализовать денежный контур и признание дохода по cash basis.

---

## 8.2 Обязательные сущности

- `Payment`
- `CashOperation`
- `FinanceEntry`
- `Expense`
- `MarketingExpense`

---

## 8.3 Обязательные функции

- создание оплаты с `Idempotency-Key`
- фиксация денежной операции
- признание дохода в finance только на базе оплаты
- базовый учёт расходов
- отображение связки order ↔ payment ↔ finance entry
- события `payment.completed`, `finance.revenue_recognized`, `finance.expense_recorded`

---

## 8.4 Критические ограничения

- order не хранит оплату как первичный факт
- доставка не создаёт доход без оплаты
- повторная оплата по тому же ключу не должна создавать второй факт

---

## 8.5 Done criteria

- денежные операции идемпотентны
- revenue признаётся строго по оплате
- order показывает оплату как производную информацию
- finance можно сверить с payments

---

## 9. Этап 6. Logistics + Fulfillment MVP

## 9.1 Цель

Привязать исполнение заказа к логистическому факту, а не к менеджерскому действию.

---

## 9.2 Обязательные сущности

- `DeliverySlot`
- `PickupWindow`
- `Driver`
- `Vehicle`
- `RouteDay`
- `DeliveryTask`
- расширение `Fulfillment`

---

## 9.3 Обязательные функции

- планирование доставки
- назначение задачи
- подтверждение исполнения доставки
- самовывоз через окно выдачи
- публикация `delivery.task_created`, `delivery.assigned`, `delivery.delivered`, `pickup.issued`, `delivery.failed`, `delivery.rescheduled`
- запуск расхода товара по факту исполнения

---

## 9.4 Критические ограничения

- `Delivered -> Planned` запрещён
- доставка без `DeliveryTask` запрещена
- расход товара без факта исполнения запрещён

---

## 9.5 Done criteria

- логистика живёт в своём домене
- исполнение доставки подтверждается отдельно
- складской расход запускается только после исполнения
- order-progress и delivery-progress различимы

---

## 10. Этап 7. Returns + Reconciliation + Audit Hardening

## 10.1 Цель

Сделать систему устойчивой к реальным операционным отклонениям: возвратам, ручным вмешательствам и междоменным расхождениям.

---

## 10.2 Обязательные сущности

- `ReturnRequest`
- `AuditEvent`
- `ReconciliationReport`

---

## 10.3 Обязательные функции

- создание возврата только через `ReturnRequest`
- approve / reject / process / close возврата
- возврат товара на склад или списание
- возврат денег с обязательной ссылкой на `ReturnRequest`
- ежедневная междоменная сверка
- уведомление о расхождениях
- admin override с обязательным audit trail

---

## 10.4 Критические ограничения

- refund без `ReturnRequest` запрещён
- возврат товара без `ReturnRequest` запрещён
- закрытие возврата до обработки последствий запрещено
- override без аудита запрещён

---

## 10.5 Done criteria

- возвраты проходят единый контур
- все критические вмешательства попадают в audit
- reconciliation даёт отчёт и алертит при расхождениях
- система контролируемо переживает ошибки и ручные корректировки

---

## 11. Этап 8. KPI / Reporting / Automation Layer

## 11.1 Цель

Построить производный слой над уже работающими первичными доменами.

---

## 11.2 Обязательные функции

- live KPI
- snapshot KPI
- отчёты по продажам, оплатам, остаткам, доставкам, возвратам
- производные воронки CRM
- алерты по просроченным резервам
- алерты по зависшим заказам / доставкам / возвратам
- системные уведомления и операционные автоматизации

---

## 11.3 Критические ограничения

- KPI не становятся источником истины
- отчёты не подменяют первичные доменные таблицы
- автоматизация не должна обходить state machine и idempotency

---

## 11.4 Done criteria

- все KPI строятся из первичных фактов
- отчётность не конфликтует с reconciliation
- автоматизации используют только канонические API и события

---

## 12. Приоритет MVP

Если нужен быстрый запуск с минимально рабочей системой, минимальный порядок такой:

### MVP-1
- Foundation
- Canonical Data Model
- CRM MVP
- Orders MVP

### MVP-2
- Inventory MVP
- Payments + Finance MVP

### MVP-3
- Logistics + Fulfillment MVP
- Returns MVP

### MVP-4
- Reconciliation
- Audit hardening
- KPI / reporting / automation

Это минимальный путь, при котором проект не ломает собственную бизнес-логику.

## 13. Что можно отложить

До поздних этапов можно отложить:
- сложные дашборды
- расширенные роли и матрицу прав тонкого уровня
- интеграции со сторонними сервисами, если они не нужны для MVP
- сложные уведомления и напоминания
- BI-витрины и исторические snapshot-оптимизации
- продвинутые сценарии маршрутизации

Но нельзя откладывать:
- state machine enforcement
- idempotency критических мутаций
- cash basis
- ReturnRequest для возвратов
- аудит override
- source-of-truth discipline

## 15. Риски неправильного порядка реализации

Если порядок нарушить, система получит типовые ошибки:

### Если сначала делать KPI
- отчёты будут строиться на неканонических фактах
- потом их придётся переписывать

### Если сначала делать деньги без orders
- пропадёт корректная связка оплаты с заказом
- возрастёт риск двойных и неполных проводок

### Если делать доставку без fulfillment
- нельзя будет отделить планирование от факта исполнения

### Если делать возвраты без ReturnRequest
- система начнёт создавать разнотипные неуправляемые refund-процессы

### Если ослабить idempotency
- появятся повторные оплаты, движения и финансовые записи

### Если ослабить state machine
- UI начнёт переводить сущности в недопустимые состояния

## 16. Правило внедрения в код

Каждый новый модуль должен проходить проверку:
- какой домен является source of truth
- какие сущности создаются первично
- какие статусы допустимы
- нужны ли command endpoints
- нужен ли `Idempotency-Key`
- какие события публикуются
- какие downstream-реакции допустимы
- нужна ли сверка с другим доменом

Если на эти вопросы нет ответа, модуль не считается готовым к реализации.
